ReplicatedRegistry

A powerful two-way client-server data synchronization module for Roblox that enables efficient state management and replication across the network boundary.

Overview

ReplicatedRegistry provides a robust solution for managing shared state between clients and servers in Roblox games. It handles change tracking, delta updates, filtering, and bidirectional communication while maintaining data integrity.

Key Features

  • Two-way synchronization: Changes can flow from server to client(s) or client to server
  • Delta updates: Only changed values are transmitted over the network
  • Flexible filtering: Built-in and custom filters for access control and validation
  • Type-safe: Full TypeScript-style type annotations for better IDE support
  • Serialization support: Optional custom serialization for optimized network traffic
  • Change tracking: Automatic diffing and change detection
  • Proxy interface: Convenient API for reading and writing nested values

Quick Start

Server Setup

local ReplicatedRegistry = require(path.to.ReplicatedRegistry)

local requestFullRF = game.ReplicatedStorage.RequestFull
local sendChangesRE = game.ReplicatedStorage.SendChanges
ReplicatedRegistry.server.set_remote_instances(requestFullRF, sendChangesRE)

-- Register a table
local playerData = {
    coins = 100,
    level = 1,
    inventory = {}
}
ReplicatedRegistry.server.register("player_123", playerData)

-- Make changes and replicate
playerData.coins = 150
ReplicatedRegistry.server.to_all_clients("player_123")

Client Setup

local ReplicatedRegistry = require(path.to.ReplicatedRegistry)

-- Set up remote instances (same ones as server)
ReplicatedRegistry.client.set_remote_instances(requestFullRF, sendChangesRE)

-- View the replicated data
local playerData = ReplicatedRegistry.client.view("player_123")
print(playerData.coins) -- 150

-- Listen for changes
ReplicatedRegistry.on_recieve("player_123", function(sender, tbl, changes)
    print("Data updated!", tbl.coins)
end)

Basic Concepts

  • Register Key: A unique identifier for each replicated table
  • Registree: The internal representation of a registered table with metadata
  • Changes: An array of value/path pairs representing modifications
  • Filter: A function that validates incoming changes
  • Proxy: A convenient interface for modifying nested table values

Getting Started Guide

This guide will walk you through setting up and using ReplicatedRegistry in your Roblox game.

Prerequisites

  • Basic understanding of Roblox Lua scripting
  • Familiarity with RemoteEvents and RemoteFunctions

Step 1: Create Remote Instances

Create a folder in ReplicatedStorage for your remotes:

-- In ReplicatedStorage, create a script that runs once
local remotes = Instance.new("Folder")
remotes.Name = "ReplicationRemotes"
remotes.Parent = game.ReplicatedStorage

local requestFull = Instance.new("RemoteFunction")
requestFull.Name = "RequestFull"
requestFull.Parent = remotes

local sendChanges = Instance.new("RemoteEvent")
sendChanges.Name = "SendChanges"
sendChanges.Parent = remotes

Step 2: Server Setup

Create a server script to register and manage replicated data:

-- ServerScriptService/DataReplication.server.lua
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local Players = game:GetService("Players")

local ReplicatedRegistry = require(ReplicatedStorage.ReplicatedRegistry)
local remotes = ReplicatedStorage.ReplicationRemotes

-- Initialize the server-side remotes
ReplicatedRegistry.server.set_remote_instances(
    remotes.RequestFull,
    remotes.SendChanges
)

-- Example: Register player data
Players.PlayerAdded:Connect(function(player)
    local playerData = {
        coins = 100,
        level = 1,
        experience = 0,
        inventory = {},
        stats = {
            strength = 10,
            speed = 10
        }
    }
    
    -- Register with the player's UserId as the key
    ReplicatedRegistry.server.register(
        tostring(player.UserId),
        playerData
    )
    
    -- Notify the player's client
    ReplicatedRegistry.server.to_clients(
        tostring(player.UserId),
        {player}
    )
end)

-- Example: Update and replicate data
local function addCoins(player, amount)
    local key = tostring(player.UserId)
    local data = ReplicatedRegistry.server.view(key)
    
    data.coins += amount
    
    -- Send changes to the specific player
    ReplicatedRegistry.server.to_clients(key, {player})
end

-- Clean up when player leaves
Players.PlayerRemoving:Connect(function(player)
    -- Save data, remove from registry, etc.
end)

Step 3: Client Setup

Create a client script to view and interact with replicated data:

-- StarterPlayer/StarterPlayerScripts/DataClient.client.lua
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local Players = game:GetService("Players")

local ReplicatedRegistry = require(ReplicatedStorage.ReplicatedRegistry)
local remotes = ReplicatedStorage.ReplicationRemotes
local player = Players.LocalPlayer

-- Initialize the client-side remotes
ReplicatedRegistry.client.set_remote_instances(
    remotes.RequestFull,
    remotes.SendChanges
)

-- View the player's data
local myKey = tostring(player.UserId)
local myData = ReplicatedRegistry.client.view(myKey)

print("My coins:", myData.coins)
print("My level:", myData.level)

-- Listen for changes
ReplicatedRegistry.on_recieve(myKey, function(sender, tbl, changes)
    print("Data updated!")
    print("Coins:", tbl.coins)
    
    -- Update UI based on changes
    for _, change in changes do
        local path = table.concat(change.p, ".")
        print(string.format("Changed: %s = %s", path, tostring(change.v)))
    end
end)

Step 4: Basic Operations

Modifying Data (Server)

-- Direct modification
local data = ReplicatedRegistry.server.view("player_123")
data.coins = 500
data.stats.strength = 15

-- Send changes to all clients
ReplicatedRegistry.server.to_all_clients("player_123")

-- Or send to specific clients
ReplicatedRegistry.server.to_clients("player_123", {player1, player2})

Using the Proxy Interface (Server)

local proxy = ReplicatedRegistry.server.view_as_proxy("player_123")

-- Set nested values easily
proxy.set("coins")(500)
proxy.set("stats", "strength")(15)

-- Increment values
proxy.incr("level")(1)
proxy.incr("experience")(50)

-- Get values
local currentCoins = proxy.get("coins")

-- Replicate to all clients
proxy.replicate()

-- Or replicate to specific player
proxy.replicate(player)

Modifying and Sending from Client

-- Client-side modification (if server accepts it)
local proxy = ReplicatedRegistry.client.view_as_proxy(myKey)

-- Make local changes
proxy.set("settings", "volume")(0.8)
proxy.set("settings", "graphics")("High")

-- Send to server
proxy.replicate()

Step 5: Adding Filters (Server)

Protect your data with filters:

-- Create a filter that only accepts positive values
local positiveFilter = function(sender, path, value)
    if type(value) == "number" then
        return value >= 0
    end
    return true
end

-- Register with filter
ReplicatedRegistry.server.register(
    "player_123",
    playerData,
    positiveFilter
)

-- Or use built-in filters
local filter = ReplicatedRegistry.get_filter({
    rate_limit = 10, -- Max 10 changes per second
    player_whitelist = {123456, 789012},
    custom = function(sender, path, value)
        -- Only allow changes to "settings"
        return path[1] == "settings"
    end
})

ReplicatedRegistry.server.register("player_123", playerData, filter)

API Reference

Complete API documentation for ReplicatedRegistry.

Types

Table

export type Table = {[any]: any}

A standard Lua table with any keys and values.

FrozenTable

export type FrozenTable = {[never]: any}

A table type indicating read-only access (for type checking).

Filter

export type Filter = (sender: Player?, path: {any}, value: any) -> boolean

A function that validates changes. Returns true to accept, false to reject.

TableChanges

export type TableChanges = {{v: any, p: {any}}}

An array of changes where each change has:

  • v: The new value
  • p: The path to the value (array of keys)

RegistreeInterface

export type RegistreeInterface<T, ReplicateArgs...> = {
    set: (...any) -> (any) -> (),
    get: (...any) -> any,
    incr: (...any) -> (number) -> (),
    replicate: (ReplicateArgs...) -> (),
    full: () -> T & FrozenTable,
}

The proxy interface for convenient table manipulation.

Core Functions

on_recieve

function on_recieve(
    register_key: string,
    fn: (sender: Player?, tbl: Table, changes: TableChanges) -> ()
) -> RBXScriptConnection

Connects a callback to be invoked when changes are received for a registered table.

Parameters:

  • register_key: The unique identifier for the table
  • fn: Callback function receiving:
    • sender: The Player who sent changes (nil if from server)
    • tbl: The updated table
    • changes: Array of applied changes

Returns: RBXScriptConnection that can be disconnected

Example:

local conn = ReplicatedRegistry.on_recieve("player_data", function(sender, tbl, changes)
    print("Received", #changes, "changes")
    for _, change in changes do
        print("Path:", table.concat(change.p, "."), "Value:", change.v)
    end
end)

get_filter

function get_filter(nameToArgs: FilterList) -> Filter

Creates a composite filter from multiple named filters.

Parameters:

  • nameToArgs: Table mapping filter names to their arguments

Returns: A combined filter function

Example:

local filter = ReplicatedRegistry.get_filter({
    player_whitelist = {123456, 789012},
    rate_limit = 10,
    custom = function(sender, path, value)
        return value >= 0
    end
})

set_change_serde

function set_change_serde(
    serializedType: string?,
    ser: ((changes: TableChanges) -> any)?,
    des: ((serialized: any) -> TableChanges)?
)

Sets custom serialization/deserialization for changes sent over network.

Important: Must be called on both client and server with matching functions.

Parameters:

  • serializedType: The type returned by serializer (cannot be "table")
  • ser: Serialization function
  • des: Deserialization function

Example:

-- Using buffer for compact serialization
ReplicatedRegistry.set_change_serde(
    "buffer",
    function(changes)
        return encodeChanges(changes)
    end,
    function(encoded)
        return decodeChanges(encoded)
    end
)

get_changes

function get_changes(registerKey: any) -> TableChanges

Calculates all pending changes since last commit.

Parameters:

  • registerKey: The register key identifier

Returns: Array of changes

commit_changes

function commit_changes(registerKey: any, changes: TableChanges?)

Commits changes to the internal copy, marking them as synchronized.

Parameters:

  • registerKey: The register key identifier
  • changes: Optional pre-calculated changes (if nil, will calculate)

revert_changes

function revert_changes(registerKey: any, changes: TableChanges?)

Reverts the table to its last committed state.

Parameters:

  • registerKey: The register key identifier
  • changes: Optional pre-calculated changes (if nil, will calculate)

Filters Guide

Filters are powerful tools for controlling which changes are accepted when data is replicated. This guide covers built-in filters and how to create custom ones.

Overview

A filter is a function with the signature:

function(sender: Player?, path: {any}, value: any) -> boolean
  • sender: The player who sent the change (nil if from server)
  • path: Array of keys leading to the changed value (e.g., {"stats", "strength"})
  • value: The new value being set
  • Returns: true to accept the change, false to reject it

Built-in Filters

Player Blacklist (Server Only)

Reject changes from specific players by their UserId.

local filter = ReplicatedRegistry.get_filter({
    player_blacklist = {123456, 789012, 345678}
})

ReplicatedRegistry.server.register("game_data", data, filter)

Use Cases:

  • Temporarily blocking players under investigation
  • Preventing changes from banned players
  • Testing without certain players

Player Whitelist (Server Only)

Only accept changes from specific players by their UserId.

local filter = ReplicatedRegistry.get_filter({
    player_whitelist = {123456, 789012}
})

ReplicatedRegistry.server.register("admin_panel", data, filter)

Use Cases:

  • Admin-only data modifications
  • VIP-only features
  • Team-restricted updates

Rate Limiting

Limit how many changes can be accepted per second.

local filter = ReplicatedRegistry.get_filter({
    rate_limit = 10 -- Max 10 changes per second per sender
})

ReplicatedRegistry.server.register("player_input", data, filter)

Parameters:

  • Number: Maximum changes per second

Use Cases:

  • Preventing spam
  • Anti-exploit protection
  • Bandwidth management

Important Notes:

  • Rate limiting is per-sender (per player on server, or "server" on client)
  • Rejected changes are silently dropped
  • Consider user experience when setting limits

No Receive

Completely block incoming changes from the other side.

local filter = ReplicatedRegistry.get_filter({
    no_recieve = true
})

ReplicatedRegistry.server.register("server_only", data, filter)

Use Cases:

  • One-way replication (server → clients only)
  • Read-only data from client perspective
  • Preventing any client modifications

Custom Filters

Basic Custom Filter

local function onlyPositiveNumbers(sender, path, value)
    if type(value) == "number" then
        return value >= 0
    end
    return true -- Allow non-numbers
end

ReplicatedRegistry.server.register("player_stats", data, onlyPositiveNumbers)

Path-Based Filtering

Restrict which parts of the table can be modified:

local function onlySettings(sender, path, value)
    -- Only allow changes to the "settings" subtable
    return path[1] == "settings"
end

local function readOnlyStats(sender, path, value)
    -- Block any changes to "stats"
    return path[1] ~= "stats"
end

Value Type Validation

local function validateTypes(sender, path, value)
    local fieldName = path[#path]
    
    if fieldName == "coins" or fieldName == "level" then
        return type(value) == "number" and value >= 0
    end
    
    if fieldName == "name" then
        return type(value) == "string" and #value <= 20
    end
    
    if fieldName == "inventory" then
        return type(value) == "table"
    end
    
    return true
end

Range Validation

local function enforceRanges(sender, path, value)
    local key = path[#path]
    
    -- Stats must be between 1 and 100
    if key == "strength" or key == "speed" or key == "defense" then
        return type(value) == "number" and value >= 1 and value <= 100
    end
    
    -- Level must be between 1 and 50
    if key == "level" then
        return type(value) == "number" and value >= 1 and value <= 50
    end
    
    return true
end

Player-Specific Permissions

local function ownershipFilter(sender, path, value)
    if not sender then return true end -- Allow server
    
    -- Players can only modify their own data
    local dataOwner = path[1] -- Assuming first key is userId
    return tostring(sender.UserId) == tostring(dataOwner)
end

Anti-Cheat Filter

local lastValues = {}

local function antiCheatFilter(sender, path, value)
    if not sender then return true end
    
    local key = table.concat(path, ".")
    local playerId = sender.UserId
    
    -- Detect impossible value jumps
    if type(value) == "number" then
        local lastValue = lastValues[playerId]?[key]
        if lastValue then
            local diff = math.abs(value - lastValue)
            if diff > 100 then -- Suspicious jump
                warn("Suspicious value change from", sender.Name)
                return false
            end
        end
        
        lastValues[playerId] = lastValues[playerId] or {}
        lastValues[playerId][key] = value
    end
    
    return true
end

Combining Filters

Use get_filter to combine multiple filters:

local filter = ReplicatedRegistry.get_filter({
    rate_limit = 5,
    player_whitelist = {123456, 789012},
    custom = function(sender, path, value)
        -- Additional custom logic
        if path[1] == "coins" then
            return type(value) == "number" and value >= 0
        end
        return true
    end
})

ReplicatedRegistry.server.register("premium_data", data, filter)

Execution Order:

  1. Built-in filters are checked first (rate_limit, blacklist, etc.)
  2. If all built-in filters pass, the custom filter is called
  3. If any filter returns false, the change is rejected

Advanced Patterns

Conditional Filters

local function conditionalFilter(sender, path, value)
    -- Different rules based on game state
    local gameState = ReplicatedRegistry.server.view("game_state")
    
    if gameState.round_active then
        -- Stricter rules during active gameplay
        return path[1] == "player_input"
    else
        -- More lenient in lobby
        return true
    end
end

Logging Filter

local function loggingFilter(sender, path, value)
    -- Log all changes for debugging
    local pathStr = table.concat(path, ".")
    print(string.format(
        "[CHANGE] %s: %s = %s",
        sender and sender.Name or "Server",
        pathStr,
        tostring(value)
    ))
    
    -- Still accept the change
    return true
end

Quota System

local quotas = {}
local MAX_CHANGES = 100

local function quotaFilter(sender, path, value)
    if not sender then return true end
    
    local userId = sender.UserId
    quotas[userId] = (quotas[userId] or 0) + 1
    
    if quotas[userId] > MAX_CHANGES then
        warn(sender.Name, "exceeded change quota")
        return false
    end
    
    return true
end

-- Reset quotas periodically
task.spawn(function()
    while true do
        task.wait(60) -- Reset every minute
        quotas = {}
    end
end)

Validation with External Data

local DataStore = game:GetService("DataStoreService")
local playerPermissions = DataStore:GetDataStore("Permissions")

local function permissionFilter(sender, path, value)
    if not sender then return true end
    
    local success, permissions = pcall(function()
        return playerPermissions:GetAsync(sender.UserId)
    end)
    
    if not success or not permissions then
        return false -- Reject if we can't verify
    end
    
    -- Check if player has permission for this path
    local requiredPermission = path[1]
    return table.find(permissions, requiredPermission) ~= nil
end

Best Practices

  1. Fail Secure: When in doubt, reject changes
  2. Log Rejections: Help debug why valid changes might be rejected
  3. Consider Performance: Filters run on every change
  4. Use Rate Limiting: Always include rate limiting for client-originated changes
  5. Validate Types: Always check value types before comparing
  6. Whitelist, Don't Blacklist: It's safer to explicitly allow than block

Testing Filters

-- Test your filter with various inputs
local function testFilter()
    local filter = myCustomFilter
    
    -- Test valid inputs
    assert(filter(nil, {"coins"}, 100) == true, "Should accept positive coins")
    
    -- Test invalid inputs
    assert(filter(nil, {"coins"}, -50) == false, "Should reject negative coins")
    
    -- Test edge cases
    assert(filter(nil, {"coins"}, 0) == true, "Should accept zero")
    assert(filter(nil, {"name"}, "") == true, "Should accept empty string")
    
    print("All filter tests passed!")
end

testFilter()

Common Pitfalls

  1. Forgetting nil checks: Always check if sender is nil (server-originated)
  2. Not validating types: Check type(value) before operations
  3. Over-filtering: Too restrictive filters frustrate users
  4. No error handling: Filters can crash if not defensive
  5. Performance issues: Complex filters on high-frequency changes

Player Data System

A complete player data system with persistence, validation, and replication.

Server Implementation

-- ServerScriptService/PlayerDataSystem.server.lua
local Players = game:GetService("Players")
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local DataStoreService = game:GetService("DataStoreService")

local ReplicatedRegistry = require(ReplicatedStorage.ReplicatedRegistry)
local playerDataStore = DataStoreService:GetDataStore("PlayerData")

-- Initialize remotes
local remotes = ReplicatedStorage.ReplicationRemotes
ReplicatedRegistry.server.set_remote_instances(
    remotes.RequestFull,
    remotes.SendChanges
)

-- Default player data template
local function createDefaultData()
    return {
        coins = 100,
        level = 1,
        experience = 0,
        stats = {
            strength = 10,
            speed = 10,
            defense = 10
        },
        inventory = {},
        settings = {
            music = true,
            sfx = true,
            graphics = "Medium"
        },
        lastLogin = os.time()
    }
end

-- Data validation filter
local function createPlayerDataFilter()
    return ReplicatedRegistry.get_filter({
        rate_limit = 10,
        custom = function(sender, path, value)
            if not sender then return true end
            
            -- Block modifications to coins and experience from client
            local field = path[1]
            if field == "coins" or field == "experience" or field == "level" then
                return false
            end
            
            -- Allow settings changes
            if field == "settings" then
                return true
            end
            
            -- Block everything else
            return false
        end
    })
end

-- Load or create player data
local function loadPlayerData(player: Player)
    local key = tostring(player.UserId)
    local success, data = pcall(function()
        return playerDataStore:GetAsync(key)
    end)
    
    if not success or not data then
        data = createDefaultData()
    end
    
    data.lastLogin = os.time()
    
    -- Register with filter
    ReplicatedRegistry.server.register(
        key,
        data,
        createPlayerDataFilter()
    )
    
    -- Send to player
    ReplicatedRegistry.server.to_clients(key, {player})
    
    return data
end

-- Save player data
local function savePlayerData(player: Player)
    local key = tostring(player.UserId)
    local data = ReplicatedRegistry.server.view(key)
    
    pcall(function()
        playerDataStore:SetAsync(key, data)
    end)
end

-- Player joined
Players.PlayerAdded:Connect(function(player)
    loadPlayerData(player)
end)

-- Player leaving
Players.PlayerRemoving:Connect(function(player)
    savePlayerData(player)
end)

-- Periodic auto-save
task.spawn(function()
    while true do
        task.wait(300) -- Save every 5 minutes

    round = 0,
    timeLeft = 0,

    map = "Map1"

-- Game loop
local function startRound()
    local proxy = ReplicatedRegistry.server.view_as_proxy("game_state")
    
    proxy.incr("round")(1)
    proxy.set("status")("Playing")
    proxy.set("timeLeft")(180) -- 3 minutes
    
    -- Set alive players
    local alivePlayers = {}
    for _, player in Players:GetPlayers() do
        alivePlayers[tostring(player.UserId)] = true
    end
    proxy.set("players_alive")(alivePlayers)
    
    proxy.replicate()
end

-- Update timer
RunService.Heartbeat:Connect(function(dt)
    local state = ReplicatedRegistry.server.view("game_state")
    
    if state.status == "Playing" then
        local proxy = ReplicatedRegistry.server.view_as_proxy("game_state")
        proxy.incr("timeLeft")(-dt)
        
        -- Only replicate every 0.5 seconds
        if math.floor(state.timeLeft * 2) ~= math.floor((state.timeLeft - dt) * 2) then
            proxy.replicate()
        end
        
        if state.timeLeft <= 0 then
            endRound()
        end
    end
end)

-- Client
local gameState = ReplicatedRegistry.client.view("game_state")

ReplicatedRegistry.on_recieve("game_state", function(sender, tbl, changes)
    -- Update UI based on state
    if tbl.status == "Playing" then

            math.floor(tbl.timeLeft / 60),
            math.floor(tbl.timeLeft % 60)
        )
    elseif tbl.status == "Waiting" then
        statusLabel.Text = "Waiting for players..."
    end
    
    -- Check for round changes
    for _, change in changes do
        if change.p[1] == "round" then
            print("Round", change.v, "started!")
        end
    end
end)

Inventory System

-- Server
local MAX_INVENTORY_SIZE = 20

local inventoryFilter = function(sender, path, value)
    if not sender then return true end
    
    -- Players can't directly modify their inventory
    -- Must go through server functions
    return false
end

Players.PlayerAdded:Connect(function(player)
    local key = "inventory_" .. player.UserId
    ReplicatedRegistry.server.register(key, {
        items = {},
        capacity = MAX_INVENTORY_SIZE
    }, inventoryFilter)
end)

-- Add item function
local function addItem(player: Player, itemId: string, quantity: number)
    local key = "inventory_" .. player.UserId
    local proxy = ReplicatedRegistry.server.view_as_proxy(key)
    local inventory = proxy.full()
    
    -- Check capacity
    local totalItems = 0
    for _, item in inventory.items do
        totalItems += item.quantity or 1
    end
    
    if totalItems + quantity > inventory.capacity then
        return false, "Inventory full"
    end
    
    -- Add item
    local existingIndex = nil
    for i, item in inventory.items do
        if item.id == itemId then
            existingIndex = i
            break
        end
    end
    
    if existingIndex then
        proxy.incr("items", existingIndex, "quantity")(quantity)
    else
        table.insert(inventory.items, {
            id = itemId,
            quantity = quantity,
            timestamp = os.time()
        })
        proxy.set("items")(inventory.items)
    end
    
    proxy.replicate(player)
    return true
end

-- Client
local key = "inventory_" .. player.UserId
local inventory = ReplicatedRegistry.client.view(key)

    for i, item in inventory.items do
        local slot = inventoryUI:FindFirstChild("Slot" .. i)
        if slot then
            slot.ItemIcon.Image = getItemIcon(item.id)
            slot.Quantity.Text = tostring(item.quantity)
        end
    end
end

    updateInventoryUI()

Players.PlayerAdded:Connect(function(player)

end)

Tips for All Examples

  1. Always validate on server: Never trust client data
  2. Use filters: Protect against malicious modifications
  3. Rate limit: Prevent spam and exploits
  4. Batch updates: Group related changes together
  5. Handle disconnects: Save data before players leave
  6. Error handling: Wrap DataStore calls in pcall
  7. UI updates: Only update when necessary
  8. Clear feedback: Show users when data is syncing
