Filters Guide

Filters are powerful tools for controlling which changes are accepted when data is replicated. This guide covers built-in filters and how to create custom ones.

Overview

A filter is a function with the signature:

function(sender: Player?, path: {any}, value: any) -> boolean
  • sender: The player who sent the change (nil if from server)
  • path: Array of keys leading to the changed value (e.g., {"stats", "strength"})
  • value: The new value being set
  • Returns: true to accept the change, false to reject it

Built-in Filters

Player Blacklist (Server Only)

Reject changes from specific players by their UserId.

local filter = ReplicatedRegistry.get_filter({
    player_blacklist = {123456, 789012, 345678}
})

ReplicatedRegistry.server.register("game_data", data, filter)

Use Cases:

  • Temporarily blocking players under investigation
  • Preventing changes from banned players
  • Testing without certain players

Player Whitelist (Server Only)

Only accept changes from specific players by their UserId.

local filter = ReplicatedRegistry.get_filter({
    player_whitelist = {123456, 789012}
})

ReplicatedRegistry.server.register("admin_panel", data, filter)

Use Cases:

  • Admin-only data modifications
  • VIP-only features
  • Team-restricted updates

Rate Limiting

Limit how many changes can be accepted per second.

local filter = ReplicatedRegistry.get_filter({
    rate_limit = 10 -- Max 10 changes per second per sender
})

ReplicatedRegistry.server.register("player_input", data, filter)

Parameters:

  • Number: Maximum changes per second

Use Cases:

  • Preventing spam
  • Anti-exploit protection
  • Bandwidth management

Important Notes:

  • Rate limiting is per-sender (per player on server, or "server" on client)
  • Rejected changes are silently dropped
  • Consider user experience when setting limits

No Receive

Completely block incoming changes from the other side.

local filter = ReplicatedRegistry.get_filter({
    no_recieve = true
})

ReplicatedRegistry.server.register("server_only", data, filter)

Use Cases:

  • One-way replication (server → clients only)
  • Read-only data from client perspective
  • Preventing any client modifications

Custom Filters

Basic Custom Filter

local function onlyPositiveNumbers(sender, path, value)
    if type(value) == "number" then
        return value >= 0
    end
    return true -- Allow non-numbers
end

ReplicatedRegistry.server.register("player_stats", data, onlyPositiveNumbers)

Path-Based Filtering

Restrict which parts of the table can be modified:

local function onlySettings(sender, path, value)
    -- Only allow changes to the "settings" subtable
    return path[1] == "settings"
end

local function readOnlyStats(sender, path, value)
    -- Block any changes to "stats"
    return path[1] ~= "stats"
end

Value Type Validation

local function validateTypes(sender, path, value)
    local fieldName = path[#path]
    
    if fieldName == "coins" or fieldName == "level" then
        return type(value) == "number" and value >= 0
    end
    
    if fieldName == "name" then
        return type(value) == "string" and #value <= 20
    end
    
    if fieldName == "inventory" then
        return type(value) == "table"
    end
    
    return true
end

Range Validation

local function enforceRanges(sender, path, value)
    local key = path[#path]
    
    -- Stats must be between 1 and 100
    if key == "strength" or key == "speed" or key == "defense" then
        return type(value) == "number" and value >= 1 and value <= 100
    end
    
    -- Level must be between 1 and 50
    if key == "level" then
        return type(value) == "number" and value >= 1 and value <= 50
    end
    
    return true
end

Player-Specific Permissions

local function ownershipFilter(sender, path, value)
    if not sender then return true end -- Allow server
    
    -- Players can only modify their own data
    local dataOwner = path[1] -- Assuming first key is userId
    return tostring(sender.UserId) == tostring(dataOwner)
end

Anti-Cheat Filter

local lastValues = {}

local function antiCheatFilter(sender, path, value)
    if not sender then return true end
    
    local key = table.concat(path, ".")
    local playerId = sender.UserId
    
    -- Detect impossible value jumps
    if type(value) == "number" then
        local lastValue = lastValues[playerId]?[key]
        if lastValue then
            local diff = math.abs(value - lastValue)
            if diff > 100 then -- Suspicious jump
                warn("Suspicious value change from", sender.Name)
                return false
            end
        end
        
        lastValues[playerId] = lastValues[playerId] or {}
        lastValues[playerId][key] = value
    end
    
    return true
end

Combining Filters

Use get_filter to combine multiple filters:

local filter = ReplicatedRegistry.get_filter({
    rate_limit = 5,
    player_whitelist = {123456, 789012},
    custom = function(sender, path, value)
        -- Additional custom logic
        if path[1] == "coins" then
            return type(value) == "number" and value >= 0
        end
        return true
    end
})

ReplicatedRegistry.server.register("premium_data", data, filter)

Execution Order:

  1. Built-in filters are checked first (rate_limit, blacklist, etc.)
  2. If all built-in filters pass, the custom filter is called
  3. If any filter returns false, the change is rejected

Advanced Patterns

Conditional Filters

local function conditionalFilter(sender, path, value)
    -- Different rules based on game state
    local gameState = ReplicatedRegistry.server.view("game_state")
    
    if gameState.round_active then
        -- Stricter rules during active gameplay
        return path[1] == "player_input"
    else
        -- More lenient in lobby
        return true
    end
end

Logging Filter

local function loggingFilter(sender, path, value)
    -- Log all changes for debugging
    local pathStr = table.concat(path, ".")
    print(string.format(
        "[CHANGE] %s: %s = %s",
        sender and sender.Name or "Server",
        pathStr,
        tostring(value)
    ))
    
    -- Still accept the change
    return true
end

Quota System

local quotas = {}
local MAX_CHANGES = 100

local function quotaFilter(sender, path, value)
    if not sender then return true end
    
    local userId = sender.UserId
    quotas[userId] = (quotas[userId] or 0) + 1
    
    if quotas[userId] > MAX_CHANGES then
        warn(sender.Name, "exceeded change quota")
        return false
    end
    
    return true
end

-- Reset quotas periodically
task.spawn(function()
    while true do
        task.wait(60) -- Reset every minute
        quotas = {}
    end
end)

Validation with External Data

local DataStore = game:GetService("DataStoreService")
local playerPermissions = DataStore:GetDataStore("Permissions")

local function permissionFilter(sender, path, value)
    if not sender then return true end
    
    local success, permissions = pcall(function()
        return playerPermissions:GetAsync(sender.UserId)
    end)
    
    if not success or not permissions then
        return false -- Reject if we can't verify
    end
    
    -- Check if player has permission for this path
    local requiredPermission = path[1]
    return table.find(permissions, requiredPermission) ~= nil
end

Best Practices

  1. Fail Secure: When in doubt, reject changes
  2. Log Rejections: Help debug why valid changes might be rejected
  3. Consider Performance: Filters run on every change
  4. Use Rate Limiting: Always include rate limiting for client-originated changes
  5. Validate Types: Always check value types before comparing
  6. Whitelist, Don't Blacklist: It's safer to explicitly allow than block

Testing Filters

-- Test your filter with various inputs
local function testFilter()
    local filter = myCustomFilter
    
    -- Test valid inputs
    assert(filter(nil, {"coins"}, 100) == true, "Should accept positive coins")
    
    -- Test invalid inputs
    assert(filter(nil, {"coins"}, -50) == false, "Should reject negative coins")
    
    -- Test edge cases
    assert(filter(nil, {"coins"}, 0) == true, "Should accept zero")
    assert(filter(nil, {"name"}, "") == true, "Should accept empty string")
    
    print("All filter tests passed!")
end

Common Pitfalls

  1. Forgetting nil checks: Always check if sender is nil (server-originated)
  2. Not validating types: Check type(value) before operations
  3. Over-filtering: Too restrictive filters frustrate users
  4. No error handling: Filters can crash if not defensive
  5. Performance issues: Complex filters on high-frequency changes
